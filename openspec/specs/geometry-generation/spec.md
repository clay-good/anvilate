# Geometry Generation Specification

## Purpose

Geometry generation deterministically turns a Design Spec IR plus generated parametric code into a valid B-Rep solid with semantic face/edge tags. Generation is pattern-constrained: the LLM selects and parameterizes audited archetypes rather than writing freestyle kernel code. The generated script is the model; the B-Rep is a build artifact.

## Requirements

### Requirement: Pattern-constrained code generation

Part code SHALL be generated by selecting and parameterizing archetypes from the audited pattern library (plate-with-holes, L-bracket, gusset, boss, rib, flange, enclosure shell, shaft, pulley mount, …); freestyle code generation MUST be disabled by default and available only behind an explicit power-user flag that is recorded in build provenance.

#### Scenario: Golden-path part uses patterns

- **WHEN** a bracket-class spec is built with default settings
- **THEN** the generated script is composed exclusively of pattern-library operations with parameters inside each pattern's declared bounds

#### Scenario: Freestyle is opt-in and traceable

- **WHEN** a user enables freestyle generation
- **THEN** the build proceeds with unrestricted code generation
- **AND** the evidence bundle and exported file metadata record that freestyle mode was used

### Requirement: Generated code gate before execution

Before execution, generated code MUST pass lint and static safety screening, and after execution the resulting shape MUST pass solid-validity checks (closed shell, no self-intersection, positive volume) before any downstream stage consumes it.

#### Scenario: Lint failure short-circuits

- **WHEN** generated code fails the lint gate
- **THEN** it is returned to the generator with diagnostics, without being executed

#### Scenario: Invalid solid rejected

- **WHEN** executed code produces an open shell or self-intersecting shape
- **THEN** the build iteration is marked failed with a geometry-validity diagnosis
- **AND** the invalid shape is never passed to validation or export

### Requirement: Sandboxed execution

Generated code SHALL execute only inside a resource-limited sandbox: no network access, filesystem restricted to a scratch directory, and enforced CPU, memory, and wall-clock caps.

#### Scenario: Network attempt blocked

- **WHEN** generated code attempts a network call
- **THEN** the call fails inside the sandbox and the iteration is marked failed with the violation logged

#### Scenario: Runaway build capped

- **WHEN** generated code exceeds its CPU or time budget
- **THEN** the process is terminated and reported as a budget failure, leaving the host system unaffected

### Requirement: Semantic tagging at creation time

Every functional face and edge SHALL be tagged with a semantic name at feature-creation time (e.g., `mount_face`, `motor_pilot_bore`), and all downstream references (loads, boundary conditions, DFM rules, drawing dimensions) MUST use tags, never kernel-assigned indices.

#### Scenario: Loads bind to tags

- **WHEN** the validation engine applies the motor load case
- **THEN** the load is applied to the faces tagged `motor_mount_pattern`, resolved on the freshly regenerated geometry

#### Scenario: No index references

- **WHEN** any subsystem requests geometry by a raw kernel index (e.g., `Face7`)
- **THEN** the request is rejected as a contract violation

### Requirement: Tag propagation through downstream operations

When a tagged face is split, merged, or consumed by a subsequent boolean or fillet operation, its successor faces SHALL inherit the tag according to documented propagation rules; load- and DFM-critical faces MUST originate from pattern primitives whose implementations guarantee tag survival.

#### Scenario: Fillet preserves tags

- **WHEN** a fillet operation splits a tagged mounting face
- **THEN** all resulting faces that derive from the original carry the original tag
- **AND** a regression test exists for each propagation rule

#### Scenario: Lost critical tag fails the build

- **WHEN** a tag referenced by a load case cannot be resolved on the regenerated geometry
- **THEN** the build fails with a tag-resolution error rather than silently applying the load elsewhere

### Requirement: Deterministic regeneration

Given the same spec, generated code, seed, and pinned kernel versions, regeneration SHALL produce an identical solid; geometry is regenerated from scratch each iteration so no stored topological references can go stale.

#### Scenario: Iteration replay

- **WHEN** the user scrubs to iteration 3 in the timeline
- **THEN** replaying iteration 3's spec and code reproduces the identical solid and scorecard

### Requirement: Pattern library contribution contract

Every pattern in the library SHALL ship with: a parametric implementation, semantic tags, declared parameter bounds, a default DFM profile, analytical check bindings, and at least 5 golden-file tests; community contributions MUST meet the same contract to merge.

#### Scenario: Contribution without tests rejected

- **WHEN** a community pattern is submitted without golden-file tests or tag declarations
- **THEN** CI rejects the contribution with the missing contract items enumerated

### Requirement: Interface detection from mating STEP files

The system SHALL deterministically analyze an imported mating-part STEP file, detect planar faces and hole patterns, and offer them to the user as interface candidates; accepted candidates become interface contracts in the spec.

#### Scenario: Bolt-pattern detection

- **WHEN** the user drops a STEP file containing a face with 4 coplanar equal-diameter holes
- **THEN** the system offers "bolt to these 4 holes?" with the detected pattern geometry
- **AND** acceptance records the pattern as a user-confirmed interface with file provenance

#### Scenario: Detection is LLM-free

- **WHEN** interface detection runs
- **THEN** no LLM is invoked; the analysis is deterministic geometry processing
